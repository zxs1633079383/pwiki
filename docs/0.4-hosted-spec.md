# pwiki 0.4 — Hosted (LiveSync-as-a-Service) — Architecture Spec

> **Status:** SPEC ONLY. Code implementation is gated on the OSS validation
> milestone defined in `~/Documents/Obsidian Vault/opportunities/pwiki.md`:
> **GitHub stars ≥ 300 by week 4 (2026-05-24 decision day)**. Building hosted
> SaaS before product-market validation on the free tier is a documented
> anti-pattern (see /opportunities/pwiki.md "death conditions").
>
> This document captures the architecture so we don't lose context between
> validation and implementation.

## Goal

Make multi-device LiveSync stop being a CouchDB project. Today users have to:
1. install Docker
2. run `obsidian-livesync` CouchDB image
3. configure CORS / `_users` / per-DB security
4. (for mobile) expose CouchDB through Cloudflare Tunnel or a VPS
5. configure the LiveSync plugin's "Setup URI"

That's a 30-minute path with at least three places to mistype credentials. **0.4
collapses it to one URL paste in the LiveSync plugin** — same URL for every
device.

## Pricing

- **$9/month** — Hosted Pro: a managed CouchDB tenant, WebDAV/Tunnel-style sync URL, daily backup
- **$29/seat/month** (≥ 5 seats) — Team Pro: shared team vault namespace, RBAC, aggregated team brief

The OSS CLI + `pwiki serve` stay free forever. Hosted is exclusively about
"replace your CouchDB ops with our credit card".

## Components

### 1. Marketing (static)
- `pwiki.dev` — Next.js or even plain HTML; one CTA: "Sign in with GitHub"
- Hosted on Cloudflare Pages (free tier).

### 2. App backend
- **Stack:** Python (FastAPI, sharing pwiki's stack) or Node (Next.js).
  Pick FastAPI to dogfood pwiki's stack.
- **Auth:** GitHub OAuth (the user is technical; passwordless wins).
- **DB:** Postgres for orgs/users/plans/sub_url; Stripe for billing.
- **Routes:**
  - `POST /signup`              — provision tenant after Stripe checkout
  - `GET  /dashboard`           — show LiveSync Setup URI + last-sync stats
  - `POST /reset-password`      — rotate per-tenant CouchDB password
  - `DELETE /account`           — deprovision (+ S3 export)

### 3. CouchDB pool (the heart)
- **Provisioning model:** one CouchDB *database* per tenant (not one CouchDB
  *instance*). All tenants share a CouchDB cluster; per-DB security isolates them.
- **Why per-DB, not per-instance:** running 1,000 CouchDB containers is silly;
  CouchDB's `_security` already namespaces. Acceptable up to ~10K paid tenants.
- **Layout:** 3-node CouchDB cluster (q=8, n=3) on Fly.io for HA.
- **Per-tenant:**
  - DB name: `pw_<tenant_id>` (UUID, not user-controlled — prevents enumeration)
  - admin user: `pw_<tenant_id>` (NOT a CouchDB cluster admin; member of own DB only)
  - random rotated password stored encrypted in Postgres (KMS-wrapped)
  - daily snapshot → S3 with tenant-scoped IAM, 7-day retention
- **CORS:** already set for `app://obsidian.md`, `capacitor://localhost` (matches our `init-couchdb.sh` from the personal Vault setup).

### 4. The Setup URI (the user-facing magic)
The LiveSync plugin accepts a single URI:
```
obsidian://setuplivesync?settings=<base64-encoded JSON>
```

When the user clicks "Connect to pwiki" in `/dashboard`, we redirect to the
above URI with the JSON pre-filled. **One click on every device** = the goal.

The encoded settings include:
- `couchDB_URI`
- `couchDB_USER` / `couchDB_PASSWORD` (per-tenant)
- `couchDB_DBNAME`
- E2EE passphrase (we don't store this; user picks it once, written to URI)

### 5. Edge / sync layer
LiveSync plugin connects directly to `couch.pwiki.dev/{db_name}` over HTTPS.
- TLS termination at Cloudflare; pass-through to CouchDB cluster
- Cloudflare Tunnel for the cluster → no public IP needed
- WebDAV not exposed (LiveSync uses CouchDB API only)

### 6. Mobile
"Mobile users should just work" is the table-stakes promise. Path:
- iOS / Android Obsidian app + LiveSync plugin
- Same Setup URI, opened via deep link from `/dashboard` on mobile browser
- E2EE passphrase entered once per device

This is the entire reason 0.4 exists. The current `Self-hosted LiveSync` README
makes mobile setup a 5-step ordeal.

## Tradeoffs

| Decision | Pick | Why |
|---|---|---|
| Per-DB vs per-instance | per-DB | 1,000 CouchDB containers is silly |
| Auth | GitHub OAuth | technical user, passwordless wins |
| Backup | S3 + daily snapshot | 7d retention + tenant-scoped IAM |
| Billing | Stripe checkout | $9/mo SaaS standard |
| Region | US East + EU West | LiveSync needs <100ms |
| E2EE passphrase | client-only, never stored | trust model: we hold ciphertext, never plaintext |
| RBAC for Team | per-DB members + namespace | CouchDB security doc per DB |

## Anti-patterns to avoid (learned from incumbents)

- **Don't gate the OSS CLI behind a license.** Sentry / Posthog / GitLab models
  show OSS-first wins; Karpathy's gist + 6 OSS implementations make CLI
  paywalls competitively dead.
- **Don't try to host the CLI's brain.** Local-first is the wedge — hosting
  the wiki itself defeats the whole pattern. We host *sync*, not *content*.
- **Don't add an LLM API to hosted billing.** Users bring their own Claude /
  OpenAI key for `pwiki query --rag` (well, the OSS RAG is local; the hosted
  RAG has cached embeddings but still calls the user's LLM provider for
  generation if they want chat-style answers).

## Implementation order (post-traction)

1. **Week 1:** Postgres schema + GitHub OAuth + Stripe checkout (dummy CouchDB provisioning, no real one yet)
2. **Week 2:** CouchDB cluster on Fly.io, per-tenant provisioning lambda
3. **Week 3:** Setup URI generator + `/dashboard` UI; manual onboarding for first 5 alpha users
4. **Week 4:** mobile flow QA (iOS + Android); backup/restore; email notifications
5. **Week 5:** public launch + first paid tier

Total: ~5 weeks engineering after traction is validated.

## Cost model (back-of-envelope)

- Fly.io 3-node CouchDB cluster (small): ~$60/month
- Postgres (Neon free tier first, then $19/month): ~$0–$19
- Cloudflare (Tunnel + Pages + R2 backup): ~$5/month for 100 users
- Stripe fees: 2.9% + $0.30 per txn

At $9/month per user, breakeven at **15 paying users** for cloud cost.
Realistic target: 100 paying users in month 3 → $900 MRR → solid runway for 0.5+.

## Why this isn't 0.3.x

0.3 is "browse my vault locally" — a feature I could have written in v0.1 and
just split out for organization. **0.4 is a different product**: it's a
multi-device sync service. Different infra, different SLA, different billing,
different support load. Conflating them in one version would muddle both.

## Open questions (revisit at decision day)

- **Self-hosted Pro?** — Some users will want to host it themselves (compliance / sovereignty). Optional Helm chart in 0.5+? Currently no.
- **Backup encryption** — KMS-wrapped is table-stakes; what about user-controlled keys (BYOK)? Defer to post-launch demand.
- **Rate limiting** — CouchDB's built-in is weak; Cloudflare or in-app limiter? Decide week 2.
- **LLM-side billing** — if we ever offer hosted RAG-with-chat, do we proxy LLM tokens or require BYOK? Strong instinct: BYOK from day one. Margin on token resale is brutal.
